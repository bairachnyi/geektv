GitHub-режим превращает SmallTV Ultra в компактную CI/CD-панель. Устройство
показывает GitHub Actions, deployments/environments, проверки pull request и
релизы, но не хранит GitHub-токены и не обращается к GitHub API напрямую.

## Как устроен поток данных

1. GitHub App отправляет события в защищённый webhook bridge сразу после
   изменения workflow, deployment, PR checks или release.
2. Bridge проверяет подпись `X-Hub-Signature-256`, отбрасывает повторные delivery
   ID и сохраняет нормализованное состояние локально.
3. ESP8266 запрашивает `/api/github`, разбирает максимум шестнадцать событий и рисует
   по две крупные карточки на страницу.
4. Между запросами устройство самостоятельно анимирует индикаторы и увеличивает
   секундомер активных событий.
5. Если есть `queued`, `waiting`, `pending` или `in_progress`, экран переходит в
   `FOCUS`: показывает только активные события и удерживает GitHub-режим в общей
   карусели до завершения работы.

В webhook-режиме GitHub PAT не требуется. Устройство хранит лишь URL локальной
ленты и необязательный отдельный device token. Webhook secret хранится только в
`emulator/config.local.json`, а события — в игнорируемом
`emulator/github-events.local.json`.

## Все элементы экрана GH//STAT

| Элемент | Значение |
| --- | --- |
| `GH//STAT` | Название GitHub status dashboard. |
| Голубая точка и `LIVE` | Получен свежий корректный ответ bridge. Это не означает, что каждое событие активно. |
| `1/2` | Текущая страница и общее количество страниц. На странице до двух крупных карточек. |
| `R` | Количество событий в `queued`, `waiting`, `pending` или `in_progress`. |
| `P` | Количество последних событий с успешным завершением. |
| `F` | Количество событий с ошибкой. |
| `FOCUS` | Есть активная или ожидающая работа. Завершённая история временно скрыта, а GitHub-режим удерживается в общей карусели. |
| Голубое вращающееся кольцо | Workflow/deployment/check выполняется сейчас. |
| Жёлтое вращающееся кольцо | Событие ожидает запуска или свободного runner. |
| Статичная зелёная галочка | Событие успешно завершено. |
| Статичный красный круг `!` | Событие завершилось с ошибкой. |
| `MM:SS` / `HhMM` | Прошедшее время активного события. Счётчик продолжается локально между запросами. |
| `PASS`, `FAIL`, `STOP` | Итог завершённого события. |
| Крупное имя | Короткое имя репозитория без owner. |
| Вторая строка | Workflow, environment, номер PR или release tag. |
| `ACT` | GitHub Actions workflow run. |
| `DEP` | Deployment и его environment/status. |
| `PR` | Совокупный результат check runs открытого pull request. |
| `REL` | Последний опубликованный GitHub Release. |
| Branch/ref | Ветка, deployment ref или release target. |
| `22 JUL 23:03` | Дата и время последнего изменения события. Формируется в локальном часовом поясе компьютера с bridge. |
| Двойная голубая рамка | Самое новое событие по времени последнего изменения GitHub. |

Для активного события дата показывает последнее известное изменение GitHub, а
счётчик — время от начала текущего workflow, deployment или check run. Это два
разных показателя.

## Поля вкладки GitHub

### GitHub deploy dashboard

| Поле | Что вводить | Ограничение/поведение |
| --- | --- | --- |
| Status feed URL | LAN-адрес вида `http://192.168.1.139:8788/api/github` | Должен начинаться с `http://` или `https://` и заканчиваться `/api/github`. `localhost` и `127.0.0.1` недоступны устройству. |
| Device token | Необязательный общий секрет bridge | Это **не** GitHub PAT. Значение должно совпадать с переменной `DEVICE_TOKEN` bridge и отправляется как `X-Device-Token`. Пустое поле сохраняет старое значение. |
| Refresh data | Частота запроса устройства к bridge | 5–3600 секунд. Обычно 15–30 секунд. |
| Rotate pages | Время показа страницы | 3–300 секунд. На странице два события. Во время `FOCUS` листаются только активные карточки. |

### GitHub accounts & repositories

| Поле | Что означает |
| --- | --- |
| Bridge mode: Mock data | Никаких запросов к GitHub. Используются демонстрационные данные и сценарии ошибок. |
| Bridge mode: Live GitHub | Показывает реальные события из выбранного delivery-механизма. |
| Live delivery: GitHub webhooks | Рекомендуемый режим. GitHub немедленно отправляет события, обычное обновление экрана не расходует REST quota. |
| Live delivery: REST polling | Совместимый резервный режим с API cache и rate-limit protection. |
| Public webhook delivery URL | Постоянный публичный HTTPS URL, доступный GitHub и пересылающий неизменённые body/headers в `/api/github/webhook`. |
| Webhook secret | Случайный секрет длиной не менее 20 символов. Пустое поле сохраняет записанное значение. |
| GitHub Actions | Принимает `workflow_run`. |
| Deployments & environments | Принимает `deployment` и `deployment_status`. |
| Pull request checks | Принимает `pull_request` и совокупный `check_suite`. |
| Releases | Принимает опубликованные `release`. |
| Owner | Фильтр owner, например `bairachnyi` или `ananas-it`. В webhook-режиме PAT рядом необязателен. |
| Fine-grained token | Нужен только для REST polling или будущего восстановления состояния через API. |
| Repository allowlist | Пустое поле принимает все repositories указанных owners. Заполните список только для ограничения экрана. |
| Bridge cache / GitHub refresh | Используется только REST polling. На webhook delivery не влияет. |

Кнопки:

- **Load bridge** — читает публичную конфигурацию bridge. Секретные токены не
  возвращаются; UI показывает только факт их наличия.
- **Save bridge** — проверяет поля и сохраняет их на компьютере с bridge.
- **Test feed** — запрашивает ленту тем же способом, что и устройство, и выводит
  конкретную ошибку.

## Создание GitHub App

GitHub App удобнее отдельных repository webhooks: одна установка получает
события всех repositories, к которым ей выдан доступ, включая новые проекты.

1. Сначала подготовьте постоянный публичный HTTPS URL. Он должен передавать
   запрос без изменения body и заголовков в
   `http://<LAN-IP>:8788/api/github/webhook`.
2. В настройках bridge выберите **GitHub webhooks**, нажмите **Generate**,
   скопируйте secret и сохраните настройки.
3. Откройте `https://github.com/settings/apps/new` и создайте private GitHub App.
4. Включите webhooks, вставьте публичный URL и тот же secret.
5. Выдайте только read-only repository permissions: **Actions**, **Checks**,
   **Contents**, **Deployments**, **Pull requests**.
6. Подпишитесь на `workflow_run`, `deployment`, `deployment_status`,
   `pull_request`, `check_suite`, `release`.
7. Установите App с вариантом **All repositories** отдельно на `bairachnyi` и
   `ananas-it`. Для организации установка может потребовать подтверждения owner.

В GitHub нельзя создать один user-level webhook для всех личных repositories,
поэтому GitHub App предпочтительнее ручного добавления webhook в каждый проект.

Bridge отвечает `202 Accepted` сразу после проверки подписи, использует
`X-GitHub-Delivery` для защиты от повторов и хранит до 1000 текущих/недавних
событий. Завершённая история удаляется через 14 дней.

## Ограничения GitHub API и частота обновления

GitHub разрешает 60 REST-запросов в час без аутентификации и обычно 5 000
запросов в час для запросов с PAT. Одно обновление GH//STAT — это не один
запрос: при включённых Actions, Deployments, PR checks и Releases требуется до
семи запросов на каждый repository, плюс запросы автоматического поиска.

Поэтому интервал 10 секунд допустим между устройством и локальным bridge, но не
между bridge и GitHub. Bridge:

- объединяет одновременные запросы двух страниц и устройства в один цикл;
- выполняет GitHub-запросы последовательно;
- автоматически увеличивает cache до безопасного значения с запасом;
- при `RATE_LIMITED` соблюдает `Retry-After` или `X-RateLimit-Reset`;
- использует первый сохранённый PAT для аутентификации публичных запросов к
  owner без собственного токена.

Последний пункт не открывает приватные repositories организации. Для них нужен
отдельный fine-grained PAT с выбранным resource owner и, если требуется,
одобрением администратора организации.

Webhook delivery не расходует REST quota при обычной работе и не зависит от
количества установленных repositories. Устройство может читать локальную ленту
каждые 5–10 секунд; событие появится на ближайшем таком обновлении.

## Запуск bridge и эмулятора

```bash
node emulator/server.mjs
```

Терминал напечатает два адреса:

- `http://localhost:8788` — браузерный эмулятор на этом компьютере;
- `http://<LAN-IP>:8788/api/github` — URL для SmallTV.

Bridge слушает `0.0.0.0:8788`, поэтому компьютер и SmallTV должны находиться в
одной сети, а firewall должен разрешать входящие подключения к Node.js.

Настройки bridge сохраняются в игнорируемом Git файле
`emulator/config.local.json` с правами `0600`. Не добавляйте этот файл в Git.

## Сценарии эмулятора

| Кнопка | Что проверяет |
| --- | --- |
| `idle` | Только завершённые успешные события. |
| `deploying` | Активные и ожидающие события, spinner и секундомер. |
| `failure` | Ошибки deployment и PR checks. |
| `mixed` | Смешанный основной сценарий и пагинация. |
| `token_error` | Недостаточные permissions или отсутствие одобрения организации. |
| `repo_error` | Репозиторий не существует либо скрыт от токена. |
| `bridge_offline` | Устройство не может связаться с bridge. |
| `stale_data` | Последний успешный ответ слишком старый. |

## Формат ленты v0.3

```json
{
  "ok": true,
  "version": "0.5.0",
  "updatedAt": "2026-07-22T16:05:57.703Z",
  "items": [
    {
      "repo": "ananas-it/web",
      "type": "deployment",
      "workflow": "production",
      "branch": "main",
      "status": "in_progress",
      "conclusion": "",
      "age": 51,
      "at": 1784736306,
      "updatedAt": 1784736361,
      "latest": true,
      "when": "22 JUL 23:03"
    }
  ]
}
```

`age` — количество секунд от начала активного события; `at` — Unix timestamp
начала, используемый bridge для обновления `age`; `updatedAt` — время последнего
изменения GitHub; `latest` включает голубую рамку у самого нового события;
`when` — готовая компактная локальная дата последнего изменения. ESP8266
принимает максимум шестнадцать элементов.
Для совместимости клиент также понимает старый массив `runs`.

Длинные названия repository, workflow/environment и branch не обрезаются
навсегда: после короткой паузы поле движется влево как циклическая бегущая строка,
проходит через небольшой пробел и снова начинается с первого символа. Короткие
строки остаются неподвижными, чтобы экран не выглядел перегруженным.

Когда GitHub включён в общий Carousel и хотя бы один процесс имеет состояние
`queued`, `waiting`, `pending`, `in_progress` или `running`, прошивка удерживает
GH//STAT на экране. После того как очередной ответ bridge подтвердит завершение
всех активных процессов, итог остаётся виден ещё один полный интервал Carousel,
затем обычное переключение тем возобновляется.

## Ошибки

| Экран/код | Причина | Что сделать |
| --- | --- | --- |
| `GITHUB NOT SET` | Не задан feed URL | Открыть GitHub tab и вставить LAN URL. |
| `BAD FEED URL` | URL неверного формата | Использовать полный `http(s)://…/api/github`. |
| `BRIDGE OFFLINE` | Bridge выключен, IP изменился или порт закрыт | Запустить bridge, проверить LAN IP и firewall. |
| `TOKEN DENIED` | Неверный PAT, недостаточно permissions, нет org approval или неверный device token | Проверить, какой именно токен отклонён, permissions и approval. |
| `REPO NOT FOUND` | Ошибка имени или нет доступа | Проверить `owner/repository` и selected repositories токена. |
| `NO REPOS` | Автопоиск не нашёл доступных repositories | Добавить allowlist либо токен нужного owner. |
| `RATE LIMITED` | Исчерпан GitHub API rate limit | Bridge сам приостанавливает запросы до времени reset. Указать repository allowlist, добавить PAT каждому private owner и отключить ненужные события. |
| `WEBHOOK SECRET MISSING` | Webhook mode включён без сохранённого секрета | Сгенерировать secret, скопировать его в GitHub App и сохранить bridge settings. |
| `WEBHOOK SIGNATURE INVALID` | GitHub App и bridge используют разные secrets или relay изменил body | Вставить одинаковый secret и пересылать raw body без изменений. |
| `GITHUB TIMEOUT` | GitHub не ответил за 12 секунд | Проверить интернет bridge и повторить позже. |
| `BAD RESPONSE` | Bridge/GitHub вернул повреждённый или несовместимый JSON | Проверить версию bridge и его журнал. |
| `DATA STALE` | Нет свежего успешного ответа дольше трёх poll-интервалов плюс 10 секунд | Проверить bridge, Wi-Fi и GitHub API. |
| `LOW MEMORY` | ESP8266 не хватает heap для HTTPS | Использовать локальный HTTP bridge вместо прямого HTTPS. |

Если один GitHub endpoint сломан, а остальные ответили, bridge передаёт первую
структурированную warning как ошибку, чтобы частичный сбой не оставался незаметным.

## Ограничения и безопасность

- Bridge должен работать постоянно, пока нужен мониторинг.
- Не открывайте порт 8788 в интернет без reverse proxy, TLS и аутентификации.
- GitHub PAT не должен попадать в firmware, `config.json`, screenshots, commits
  или логи.
- Device token защищает локальный bridge от случайных клиентов, но обычный HTTP
  в LAN не шифрует трафик.
- ESP8266 хранит до шестнадцати событий и показывает два на страницу ради читаемости
  и экономии RAM.
- Локальный порт нельзя просто открыть в интернет. Используйте постоянный HTTPS
  tunnel/reverse proxy и webhook secret.
- GitHub не выполняет автоматическую повторную доставку failed webhooks, поэтому
  bridge должен работать постоянно; пропущенную delivery можно redeliver в GitHub.
